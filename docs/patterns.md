# Modeling Readiness Patterns

A fixed set of named patterns that address modeling debt under the
Modeling Readiness Framework. Each pattern is a concrete, repeatable
implementation move that takes a specific kind of debt and reduces it.

The Modeling Readiness Assessor agent references these patterns by name
in remediation hints attached to findings, and in the remediation plan
deliverable. Patterns are referenced — not generated on the fly — so
that customers and architects encounter consistent vocabulary across
every assessment, every customer engagement, and every conversation
about Fabric IQ readiness.

## How patterns work

Each pattern has:

- A **name** (referenced exactly in agent output)
- The **discipline** it serves
- The **debt type** it addresses
- A **v1 detection mode** (how the v1 scanner flags this pattern's
  debt; see below)
- A **what** (the implementation move, in 2–3 sentences)
- A **why** (the principle the pattern enforces, in 1–2 sentences)
- A **Fabric IQ realization** (how the pattern is implemented in
  Fabric IQ specifically)
- An **anti-pattern** (what teams do instead, and why it fails)
- A **starting move** (the smallest version of the pattern a customer
  can ship in a sprint)

**v1 detection mode** is one of:

- **Auto** — the scanner detects this pattern's debt deterministically
  from semantic-model and ontology metadata. Findings are produced
  without user confirmation.
- **Signal-based** — the scanner flags this pattern's debt only when
  an unambiguous structural signal is present. When the signal is
  absent or ambiguous, the relevant scope is marked "not assessed in
  this version" rather than scored, per Principle II of the
  constitution.
- **Deferred to v2** — the v1 scanner does not detect this pattern's
  debt. The pattern remains referenceable in remediation hints when
  another pattern's finding implies it, but the agent does not flag
  it on its own.

The detection mode is tied to the framework's scope-honesty discipline:
Canonical Entity Modeling patterns are Auto; Field-Level Lineage patterns
are Auto; Layered Modeling and Steward-Loop Modeling patterns are
Signal-based or Deferred, reflecting v1's deliberately limited diagnostic
reach into those disciplines.

Patterns are not exhaustive. They cover the high-frequency debt types
the agent diagnoses in v1. As v2 expands diagnostic coverage to Layered
Modeling and Steward-Loop Modeling deep analysis, additional patterns
will be added; existing patterns will not be renamed or restructured.

## Patterns for Canonical Entity Modeling

### CEM-01: Canonical Entity Declaration

- **Discipline**: Canonical Entity Modeling
- **Debt type**: Multiple semantic models or ontologies define the same
  logical business entity inconsistently, with no designated source of
  truth.
- **v1 detection mode**: Auto — the scanner enumerates entity
  definitions across semantic models and ontologies, applies
  configurable name-similarity rules to candidate-match them, and
  emits a finding when two or more definitions of the same logical
  entity disagree on primary key, relationships, or core attributes.
- **What**: For each contested business entity (Customer, Product,
  Vendor, etc.), designate one canonical definition — chosen with the
  business owners of every system that contributes to that entity — and
  document it as the reference all other models defer to. The canonical
  definition specifies the primary key, the required attributes, the
  relationships to other canonical entities, and the rules for
  reconciling against source-system records.
- **Why**: An entity that means three different things in three
  different reports cannot ground an AI agent. The agent will pick the
  wrong definition for the wrong question, confidently. Canonicalization
  is the precondition for trustworthy reasoning over the entity.
- **Fabric IQ realization**: The canonical definition lives as an entity
  type in a designated "canonical" Fabric IQ ontology. Other ontologies
  reference this entity type via cross-ontology binding rather than
  redefining it. The canonical ontology lives in a workspace separate
  from extension workspaces (see LM-01).
- **Anti-pattern**: Letting Fabric IQ's autogenerate-from-semantic-model
  feature produce an ontology that inherits whatever definition was in
  the .pbix. The autogenerated ontology *looks* like a canonical
  definition but is actually a frozen snapshot of one model's view, and
  the inconsistency with other models is preserved silently.
- **Starting move**: Pick the single most-contested entity in the
  customer's environment. Run a one-day workshop with owners of every
  system that touches it. Produce a written canonical definition. Bind
  the canonical Fabric IQ entity type to that definition. Refactor one
  semantic model to defer to the canonical definition. The pattern is
  proven; expand from there.

### CEM-02: Reconciliation Rule Documentation

- **Discipline**: Canonical Entity Modeling
- **Debt type**: Records representing the same real-world entity exist
  in multiple source systems with no documented rule for which to trust
  when they disagree.
- **v1 detection mode**: Auto — when CEM-01 detects a canonical entity
  conflict, the scanner additionally checks whether any of the
  conflicting definitions encode reconciliation rules as ontology
  rules/constraints. Absence of reconciliation rules across all
  conflicting definitions emits a CEM-02 finding alongside the CEM-01
  finding.
- **What**: For each canonical entity, document the explicit rules that
  govern reconciliation across source systems: which system's value
  wins for which attribute, under which conditions, with what
  precedence. The rules are versioned, owned by the canonical entity's
  data steward, and reviewed when source-system schemas change.
- **Why**: Reconciliation rules are usually tribal knowledge — Sarah
  knows that the CRM customer name is authoritative except when the
  ERP one is more recent, except for B2B customers where MDG wins.
  When the rules aren't written down, every new analyst rediscovers
  them imperfectly, and an AI agent cannot follow them at all.
- **Fabric IQ realization**: Reconciliation rules are encoded as
  ontology rules and constraints, with each rule annotated by the
  steward and source-system owners who authored it. Conflicts that
  cannot be resolved by rule are surfaced as findings to the steward
  rather than silently merged.
- **Anti-pattern**: Treating reconciliation as an ETL concern handled
  inside data pipelines. ETL-layer reconciliation makes the rules
  invisible to the semantic layer and to AI agents grounded on the
  ontology, and it makes rule changes a code-change rather than a
  governance event.
- **Starting move**: Take the canonical entity from CEM-01 and write
  down its reconciliation rules in plain English. Show the document to
  the data stewards from each source system. Capture their corrections.
  This document is the v1 of the rule set; encode it in Fabric IQ rules
  in the next sprint.

### CEM-03: Inheritance-from-Autogenerate Audit

- **Discipline**: Canonical Entity Modeling
- **Debt type**: A Fabric IQ ontology was autogenerated from a Power BI
  semantic model that itself has known modeling debt, inheriting the
  debt into the AI grounding layer.
- **v1 detection mode**: Auto — the scanner reads ontology provenance
  metadata to identify ontologies marked as autogenerated, traces each
  back to its source semantic model, and emits a CEM-03 finding when
  the source semantic model itself participates in a CEM-01 conflict.
- **What**: For each ontology that was autogenerated rather than
  modeled deliberately, audit which semantic model it was derived from
  and whether that semantic model represents the canonical definition
  of its entities. If it does not, flag the ontology for redefinition
  using CEM-01, and treat current Data Agent answers grounded on it as
  provisional.
- **Why**: The autogenerate feature is seductive because it produces
  something that runs immediately. But the ontology inherits whatever
  was true about the source model — including the source model's
  ambiguous relationships, conflated entities, and unresolved
  conflicts. Grounding a Data Agent on an inherited-debt ontology
  produces confident wrong answers, which is worse than no answers.
- **Fabric IQ realization**: The audit is a deliberate review step
  before the ontology is exposed to Data Agent or agent-facing MCP
  endpoints. Ontologies that fail the audit are kept in a non-published
  state until they're remodeled per CEM-01 and CEM-02.
- **Anti-pattern**: Treating autogenerate as the path to production
  rather than as an exploration tool. Autogenerate is excellent for
  understanding what an ontology might look like; it is rarely
  appropriate as the actual production grounding for an AI agent.
- **Starting move**: List every ontology in the customer's environment
  and mark each one as "autogenerated" or "deliberately modeled." For
  the autogenerated ones, name the source semantic model and ask the
  semantic model's owner whether it represents canonical truth. The
  answers will surprise the customer in the right way.

## Patterns for Field-Level Lineage

### FLL-01: Source-Attribution Properties

- **Discipline**: Field-Level Lineage
- **Debt type**: Entity properties carry values without recording which
  source system the value came from, when it was extracted, or how
  confident the system was in the value.
- **v1 detection mode**: Auto — for each entity type in each ontology,
  the scanner checks whether source-attribution counterparts exist on
  source-derived properties (source-system identifier, source-record
  identifier, extraction timestamp, confidence score, or equivalents)
  and emits a Source-Attribution Gap finding for each property
  missing them. The scanner does not prescribe specific property
  names; it looks for the principle.
- **What**: Every entity property that originates from a source system
  (rather than being derived in-platform) carries first-class
  attribution properties: the source-system identifier, the source-
  record identifier, the extraction timestamp, and a confidence score
  where applicable. These attribution properties are part of the
  entity's schema, not a separate audit table.
- **Why**: When an AI agent reports a value, the user's first follow-up
  question is "where did that come from?" If the answer requires
  joining to an audit log that was designed for compliance rather than
  for grounding, the agent either can't answer or answers slowly.
  Source attribution as a property is the difference between an agent
  that explains itself and one that doesn't.
- **Fabric IQ realization**: Each entity type's properties include
  attribution counterparts (e.g., `customer_email` and
  `customer_email_source_system`, `customer_email_extracted_at`). The
  attribution properties are bound to the same source as the value
  properties and surfaced through the same Data Agent interface.
- **Anti-pattern**: Relying on Purview lineage as the substitute. Purview
  tracks dataset-level and column-level lineage, which is essential
  for governance and necessary for compliance — but it doesn't make
  per-record source attribution available to agents at query time. The
  two are complementary, not interchangeable.
- **Starting move**: Pick one entity type and add attribution properties
  for its three or four most-queried fields. Bind them. Ask the Data
  Agent a question about that entity. Watch the agent cite its sources.
  The customer's reaction is the reason to do the rest of the entities.

### FLL-02: Confidence-Aware Bindings

- **Discipline**: Field-Level Lineage
- **Debt type**: Bindings between source data and entity properties
  treat all values as equally trustworthy, even when reconciliation
  produced the value through a fuzzy match or a derived rule.
- **v1 detection mode**: Signal-based — the scanner flags FLL-02 only
  when binding metadata explicitly references a reconciliation,
  fuzzy-match, or derivation step (e.g., bindings backed by views or
  tables whose names or lineage indicate a merge). When reconciliation
  is happening but invisible to the metadata layer (the common case),
  the scope is marked "not assessed in this version" rather than
  scored as clean.
- **What**: Bindings that involve reconciliation, fuzzy matching, or
  derivation carry a confidence score per record. The score is
  preserved alongside the value as the value flows into the ontology
  and out to consumers. Low-confidence values are visibly flagged when
  surfaced by Data Agent or downstream consumers.
- **Why**: A Data Agent that says "the customer's email is X" without
  distinguishing between "the CRM stated it directly" and "we matched
  this customer to a record in MDG with 73% confidence and pulled the
  email from there" is a Data Agent that will be wrong in front of an
  executive. Confidence is the difference between grounded reasoning
  and false certainty.
- **Fabric IQ realization**: Confidence is an ontology property
  alongside the value, populated by the binding layer. Rules that
  consume the value can reference confidence in their conditions.
  Data Agent prompts include confidence in answers when relevant.
- **Anti-pattern**: Computing confidence at ETL time and discarding it
  before the value lands in the semantic layer. By the time the agent
  sees the value, it has been laundered into the appearance of fact.
- **Starting move**: Identify one binding where reconciliation happens
  (a customer master that merges CRM and ERP records, for example).
  Add a confidence column to the merge output. Bind it to the ontology.
  Configure one Data Agent prompt to surface confidence when asked.

### FLL-03: Time-Source Markers on Streaming Bindings

- **Discipline**: Field-Level Lineage
- **Debt type**: Time-series bindings (eventhouse streams, IoT
  telemetry, log events) flow into the ontology without recording
  which source emitted them or when the source's clock said they
  occurred — only when the platform received them.
- **v1 detection mode**: Auto — for each ontology binding sourced from
  an eventhouse / KQL / streaming source, the scanner verifies that
  both source-emission and platform-receipt timestamp properties are
  exposed, and emits a finding when only platform-receipt time is
  present.
- **What**: Every time-series binding records both the platform-receipt
  timestamp and the source-emission timestamp, plus the source-system
  identifier. Reasoning over time uses the source-emission time;
  troubleshooting uses the platform-receipt time. The two are visibly
  different where they differ.
- **Why**: When a Data Agent answers "what happened at 14:32 yesterday,"
  it should answer using the time the event happened, not the time
  Fabric finished ingesting it. The two diverge during ingestion delays,
  source-system clock drift, and replay scenarios. Conflating them
  produces subtle wrong answers under exactly the conditions where
  correctness matters most.
- **Fabric IQ realization**: Eventhouse bindings include both
  timestamps as ontology properties. Rules that depend on temporal
  ordering specify which clock they reference.
- **Anti-pattern**: Treating "ingestion time" and "event time" as
  interchangeable because they're usually close. The Data Agent will
  eventually be asked a question where the difference matters, and
  the customer's confidence is the asset that pays the cost.
- **Starting move**: For one streaming binding, add the source-emission
  timestamp as an explicit property. Surface it in one Data Agent
  prompt that asks a temporal question. Notice when the answers
  diverge from what the platform-receipt timestamp would have produced.

## Patterns for Layered Modeling

### LM-01: Canonical Workspace Separation

- **Discipline**: Layered Modeling
- **Debt type**: Canonical entity definitions and customer-specific
  extensions live in the same workspace, mutating each other and
  making upstream upgrades destructive.
- **v1 detection mode**: Signal-based — the scanner flags LM-01 only
  when it observes the unambiguous structural signal of a single
  workspace containing both ontologies that are referenced by other
  workspaces (i.e., behaving as canonical) AND ontology extensions
  that mutate those same shared entity types. When the signal is
  absent or the user has scoped the assessment to a single workspace,
  Layered Modeling is marked "not assessed in this version."
- **What**: The canonical ontology — the entity types, properties, and
  relationships representing the customer's authoritative model — lives
  in a workspace with restricted write access (the "core" workspace).
  Customer-specific or department-specific extensions live in
  downstream workspaces that import from core but never modify it.
  Promotions from extension workspaces to core are deliberate
  governance events.
- **Why**: When extensions and canonical model live together, every
  customer-specific addition becomes a potential breaking change to
  the canonical. Teams stop upgrading because upgrades are scary.
  Six months later, the customer is on a stale platform with a tangled
  ontology and no clear path forward.
- **Fabric IQ realization**: The core ontology lives in its own
  workspace with write access restricted to the data governance team.
  Other workspaces consume it via cross-workspace ontology references.
  Extensions add types and properties in their own workspace; promotion
  uses the Fabric Import/Export Batch APIs (March 2026 release).
- **Anti-pattern**: Single-workspace modeling where everyone has write
  access to everything. Convenient for the first month, painful for
  every month after.
- **Starting move**: Identify the customer's most stable, most widely-
  consumed ontology. Move it to its own workspace. Restrict write
  access. Have one consuming team adopt the cross-workspace reference
  pattern. Use the lessons to migrate the rest.

### LM-02: Extension Namespace Discipline

- **Discipline**: Layered Modeling
- **Debt type**: Customer-specific or department-specific extensions to
  canonical entities use names that collide with canonical names or
  with other extensions, producing schema sprawl that nobody can
  navigate.
- **v1 detection mode**: Signal-based — the scanner flags LM-02 only
  when it observes property additions in extension-role workspaces
  to canonical entity types, with no namespace prefix or other
  scoping convention applied. Absent multi-workspace scope, the
  pattern is marked "not assessed in this version."
- **What**: Every extension uses a namespace prefix or suffix indicating
  its scope (e.g., `customer.acme.priority_tier`,
  `dept.finance.gross_margin_local`). Namespaces are documented and
  reserved. The canonical model owns the unprefixed names; extensions
  do not.
- **Why**: Without namespacing, two extensions can independently add a
  property called `priority_tier` to the Customer entity, and a Data
  Agent grounded on the merged ontology will silently pick whichever
  one happens to load first. Namespacing makes extensions visible as
  extensions and prevents accidental shadowing.
- **Fabric IQ realization**: Property names in extension workspaces
  carry a namespace prefix matching the workspace's role. The core
  workspace forbids prefixed names; extension workspaces forbid
  unprefixed property additions to canonical entity types.
- **Anti-pattern**: Adding extension properties under canonical names
  because "the team needs it now and we'll fix the naming later." The
  fix never happens, the shadowing accumulates, and the ontology
  becomes unreasonable for AI grounding.
- **Starting move**: Pick a recent extension that added a property to
  a canonical entity. Rename it with a namespace prefix. Update the
  three or four downstream consumers that referenced it. The pattern
  is now demonstrated; require it for the next extension.

## Patterns for Steward-Loop Modeling

### SLM-01: Correction Capture as Typed Assertion

- **Discipline**: Steward-Loop Modeling
- **Debt type**: When users or stewards correct values that came from
  the data platform, the corrections are made in downstream tools
  (Excel, ticket comments, Slack messages) and never flow back into
  the canonical model.
- **v1 detection mode**: Signal-based — the scanner flags SLM-01 only
  when the ontology contains entity types whose names or schemas match
  a recognizable Correction shape (a Correction-typed entity, or a
  table named `corrections` / `overrides` / similar bound to an
  entity). Presence of the signal indicates a correction loop is at
  least partially modeled; absence is marked "not assessed in this
  version" rather than treated as a finding (correction loops can
  exist outside the platform and v1 cannot see them).
- **What**: Corrections are captured as typed assertions in a
  designated correction workspace, with the correcting user, timestamp,
  the original value, the corrected value, and the reason. These
  assertions are first-class data, not free-text notes. They feed the
  next ingest cycle's reconciliation rules.
- **Why**: Most enterprises have an enormous, unmeasured stream of
  manual corrections that represent real institutional knowledge about
  what their data should say. The corrections die in tools that don't
  feed the platform. The next ingest run rediscovers the same problems
  and corrects them again. The loop never closes; the corrections never
  compound.
- **Fabric IQ realization**: Corrections are submitted via a Power
  Automate flow or a Power App that writes to a "corrections" table
  in OneLake. The Fabric IQ ontology binds the correction table to a
  Correction entity type, and rules use the correction history to
  refine reconciliation logic in subsequent runs.
- **Anti-pattern**: Letting corrections live in dashboard-level
  override columns or ticket-system free-text. The corrections are
  invisible to the platform and invisible to the AI agent grounded on
  the ontology.
- **Starting move**: Identify one entity where stewards actively
  correct values today. Build a single Power Automate flow that
  captures their corrections to a OneLake table. Bind the table to a
  Correction entity type in the ontology. Show the steward that their
  next correction shows up in the ontology immediately.

### SLM-02: Rule Refinement from Correction Patterns

- **Discipline**: Steward-Loop Modeling
- **Debt type**: Even when corrections are captured as typed assertions
  (SLM-01), they accumulate without ever changing the rules that
  produced the wrong values in the first place.
- **v1 detection mode**: Deferred to v2 — detecting whether captured
  corrections actually feed rule refinement requires query-log and
  rule-change-history analysis, which is out of v1 scope. The pattern
  remains referenceable in remediation hints when SLM-01 has been
  detected, but the v1 scanner does not flag SLM-02 on its own.
- **What**: A periodic review (weekly, monthly, quarterly depending on
  velocity) examines accumulated corrections, identifies patterns
  ("this same kind of correction has happened 47 times"), and updates
  the underlying reconciliation rules so the corrections stop being
  necessary. The review is owned by the canonical entity's data
  steward, with engineering support.
- **Why**: Capturing corrections without learning from them is just a
  more elaborate filing system. The value of the loop comes from
  closing it — from the reconciliation rules getting better because
  the corrections taught them. Without this review, the steward is
  trapped in a cycle of correcting the same thing forever.
- **Fabric IQ realization**: The review uses a saved query against the
  Correction entity type, surfaced via Data Agent or a Power BI report.
  Rule changes are made in the canonical ontology and promoted through
  the workspace separation discipline (LM-01). The history of
  rule-change-by-correction is a property of the canonical ontology's
  governance log.
- **Anti-pattern**: Treating corrections as a bottomless backlog rather
  than as feedback signal. Corrections get logged, never reviewed, and
  the rules stay broken indefinitely.
- **Starting move**: Schedule one 30-minute review meeting. The
  steward, one engineer. Look at the last month of corrections for
  one entity. Identify the top three patterns. Pick one to address by
  changing the rule. Ship the rule change. Schedule the next review.

## How patterns appear in agent output

When the Modeling Readiness Assessor surfaces a finding, the finding
includes one or more pattern references. Pattern references look like:

> Finding: The Customer entity is defined inconsistently across 3
> semantic models (CRM_Customer, ERP_Customer, MDG_Customer), with
> different primary keys and different relationship logic to the Order
> entity.
>
> Discipline: Canonical Entity Modeling, maturity level 1
>
> Sources: Semantic models `crm-analytics-v3`, `erp-master-v2`,
> `mdg-canonical-v1`
>
> Recommended patterns: CEM-01 (Canonical Entity Declaration),
> CEM-02 (Reconciliation Rule Documentation)
>
> Starting move: See CEM-01 starting move. Approximate effort: one
> two-day workshop plus one sprint of follow-up.

The remediation plan deliverable groups findings by pattern, so the
customer can see — at the level of named patterns rather than
individual findings — which moves will produce the most maturity gain
across their environment.

## Versioning

Patterns are versioned with the Modeling Readiness Framework as a
whole. v1 includes the patterns above (3 for CEM, 3 for FLL, 2 for LM,
2 for SLM = 10 total). Future versions may add patterns; existing
patterns will not be renamed without a Framework version bump.

When the agent ships v2 with deep diagnostic coverage of Layered
Modeling and Steward-Loop Modeling, additional patterns will be added
to those disciplines. The CEM and FLL patterns are expected to remain
stable.
